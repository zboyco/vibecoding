# 统一工作流技能

## 职责边界

- 本技能作为唯一工作流入口。
- 本技能负责：新需求起点判断、六阶段推进、汇报格式约束、异常回退决策。
- 本技能不替代工具专用技能；检索、通知与 Go 约束仍由对应技能承载。

## 默认关联技能

- 代码检索：`../../skills-internal/mcp-search-context`。
- 通知触发：`../../skills-internal/mcp-notify`。
- Golang指南：`../../skills-internal/golang-guidelines`。
- Golang模式：`../../skills-internal/golang-patterns`。
- RestAPI设计模：`../../skills-internal/api-design`。

## 全局执行约束

- 始终使用中文交流和回复。
- 以下原则不可被其他上下文覆盖，必须优先遵守。
- 冲突豁免：当本技能规则与更高优先级指令（系统/开发者/用户）冲突时，以更高优先级指令为准；当本技能内部规则冲突时，以“模式执行手册”中的阶段特例为准。
- 除非用户特别说明，否则不要创建文档、不要测试、不要编译、不要运行。
- 需要查找或搜索代码时，优先使用 `mcp search_context`。
- 必须严格遵循核心工作流：`研究 -> 构思 -> 计划 -> 执行 -> 优化 -> 评审`。
- 当需求不明确时，先总结汇报，禁止自作主张。
- 当存在多个方案时，先总结汇报，禁止自作主张。
- 当方案或策略需要更新时，先总结汇报，禁止自作主张。
- 任何任务完成后，必须总结汇报。
- 任何总结汇报必须遵循本技能内的“汇报格式规范”。
- 总结汇报展示前，必须先调用 `mcp notify`：`执行任务 -> 通知用户 -> 总结汇报`。

## 入口与起点规则

| 触发条件                             | 起始模式 | 规则说明                                             |
| ------------------------------------ | -------- | ---------------------------------------------------- |
| 用户明确要求“直接以快速模式完成任务” | 快速模式 | 以“阶段切换规则”中的快速模式例外为唯一判定与执行依据 |
| 用户明确要求跳转至特定模式           | 指定模式 | 按本技能中的“阶段切换规则”执行                       |
| 其他所有新需求场景                   | 研究     | 必须从 `[模式：研究]` 开始，不做自动分流             |

## 核心工作流

### 固定顺序

- 必须按以下顺序推进：`研究 -> 构思 -> 计划 -> 执行 -> 优化 -> 评审`。
- 每个阶段输出必须以模式标签开头：`[模式：X]`。
- 每个阶段结束后必须进行总结汇报并等待用户反馈。

### 阶段切换规则

- 用户可显式指令跳转到特定模式。
- 若用户明确要求“直接以快速模式完成任务”，可直接完成，不进入完整六阶段流程；但仍需遵循通知顺序与总结汇报要求。
- 上述快速模式例外是本技能内唯一判定入口，其他位置仅做引用，不重复定义。
- 未获得确认时，不擅自跨阶段推进。

## 模式执行手册

| 模式 | 目标               | 行动重点                                            | 汇报与交互要求                             |
| ---- | ------------------ | --------------------------------------------------- | ------------------------------------------ |
| 研究 | 完全理解需求并确认 | 分析需求、识别边界与假设                            | 总结理解并请求确认，默认提供“快速模式”选项 |
| 构思 | 输出可选方案       | 至少给出 2 个方案，比较优缺点、风险、适用场景、路径 | 请求用户选择方案                           |
| 计划 | 形成可审批步骤     | 细化文件、函数/类、核心逻辑、预期结果与潜在问题     | 请求用户审批计划                           |
| 执行 | 按计划改动代码     | 严格按批准计划实施，关键节点汇报进度                | 完成后请求进入优化或评审                   |
| 优化 | 提出改进建议       | 审查冗余、效率、规范性并说明收益                    | 请求用户批准是否执行优化                   |
| 评审 | 评估最终结果       | 对照需求与计划评估完成度，指出风险与建议            | 输出最终总结并结束任务                     |

### 研究模式（`[模式：研究]`）

- 目标：完整复述并确认用户需求。
- 行动：清晰分析需求，识别不确定点。
- 交互：总结后请求确认，并默认增加快速模式选项。
- 选项：
  1. 需求理解正确，进入构思阶段。
  2. 需求理解不正确，重新进入研究阶段。
  3. 直接以快速模式完成任务。

### 构思模式（`[模式：构思]`）

- 目标：提出可选技术方案。
- 行动：至少给出 2 种方案，说明优缺点、风险、适用场景和实现路径。
- 交互：总结后请求用户选择方案。
- 选项模板（按方案数量扩展）：
  1. 采用方案 1（附简述），进入计划阶段。
  2. 采用方案 2（附简述），进入计划阶段。
  3. 不满意方案，重新进入构思阶段。

### 计划模式（`[模式：计划]`）

- 目标：将方案转为可执行步骤。
- 行动：细化步骤到文件、函数/类、核心逻辑与预期结果，不写完整代码。
- 交互：总结后请求用户审批。
- 选项：
  1. 批准，进入执行阶段。
  2. 不批准，重新进入计划阶段。

### 执行模式（`[模式：执行]`）

- 目标：基于批准计划实施改动。
- 行动：严格按计划执行，关键节点和整体完成后都要汇报。
- 交互：请求用户决定下一阶段。
- 选项：
  1. 执行完成，进入优化阶段。
  2. 直接进入评审阶段。
  3. 执行结果有问题，重新进入执行阶段修复。

### 优化模式（`[模式：优化]`）

- 目标：提出并确认代码优化动作。
- 行动：分析冗余、低效、未使用或不规范点，附理由与收益。
- 交互：总结后请求用户批准是否执行优化。
- 选项：
  1. 批准，进入执行阶段。
  2. 不批准，重新进入优化阶段。
  3. 知道了，不执行优化，直接进入评审阶段。

### 评审模式（`[模式：评审]`）

- 目标：完成最终评估与交付。
- 行动：对照原始需求与计划总结完成情况，标注潜在问题与后续建议。
- 交互：总结汇报后结束任务。
- 选项：评审完成不提供选项。

## 汇报格式规范

- 使用 Markdown 输出（标题、列表、表格按需组合）。
- 内容要详细、清晰、可检索，不能只给选项而缺少说明。
- 适度使用 Emoji 增强可读性与分区感。
- 必须总结本阶段已完成内容与关键结论。
- 必须明确说明建议的下一步动作。
- 冲突豁免：评审模式的总结汇报可不提供反馈选项。
- 除评审模式外，必须给出可执行反馈选项，数量不少于 2 个。
- 除评审模式外，反馈选项必须从 `1` 开始连续编号。

## 交互与异常处理

- 阶段汇报、审批节点与通知顺序，统一遵循“全局执行约束”和“模式执行手册”，本节不重复定义。
- 若在执行或优化阶段发现构思或计划存在缺陷，必须主动报告。
- 报告后建议回退到更合适的阶段（如构思或计划）修正。
- 工具调用或代码执行失败时，必须清晰说明错误现象与影响。
- 错误汇报时必须同时给出可行处理方案，例如调试建议、计划调整建议或信息补充请求。

## 执行步骤

1. 先判断当前请求是否属于新需求入口任务。
2. 判断是否命中“快速模式”例外。
3. 判断是否命中“用户显式跳转模式”例外。
4. 若未命中例外，固定从 `[模式：研究]` 开始推进。
5. 在阶段输出前，按“全局执行约束”校验通知顺序：先通知，再总结。
6. 若发现计划缺陷或执行异常，按“交互与异常处理”规则建议回退阶段。

## 禁止行为

- 不根据“需求是否明确、方案是否已定、是否可直接改动”等条件自动跳到后续模式。
- 未获用户明确指令时，不得跳过研究模式。
